Architecture Overview

How the browser talks to Fabric
💡 What is the Fabric Gateway?

The Fabric Gateway is a component built into every Fabric peer (v2.4+). It exposes a gRPC interface so client applications can submit transactions and query the ledger through a single connection point — without needing to talk to multiple peers or the orderer directly. This app's Node.js server uses the @hyperledger/fabric-gateway SDK to make those gRPC calls. Your browser talks to the Node.js server via regular HTTP, and the server bridges to Fabric via gRPC.

evaluate vs submit

The two ways to interact with chaincode
evaluate READ-ONLY

Sends a query to one peer. The peer simulates the chaincode locally and returns the result. Nothing is written to the ledger. No ordering, no consensus. Fast and lightweight.

Used for: GetAllAssets, ReadAsset, AssetExists

submit WRITE

Triggers the full transaction lifecycle: propose → endorse → order → commit. Permanently changes the ledger. The SDK waits until the transaction is committed before returning.

Used for: CreateAsset, TransferAsset, DeleteAsset

💡 How evaluate works

contract.evaluateTransaction('GetAllAssets') sends the request to a single peer. The peer runs the chaincode in simulation and returns the result directly. The result is never sent to the orderer — nothing hits the ledger. Perfect for reading current state.

Submit Transaction

contract.submitTransaction() — writes to ledger
💡 How submit works

contract.submitTransaction(...) triggers the full endorsement → ordering → commit lifecycle. The SDK blocks until the transaction is committed to the ledger (or throws if it fails). Every submitted transaction is permanently recorded on the blockchain — it cannot be deleted or rolled back.

Chaincode Events

network.getChaincodeEvents() — async event stream
💡 What are Chaincode Events?

When a chaincode function executes, it can call stub.SetEvent(name, payload) to emit a named event. After the transaction is committed to the ledger, the Fabric Gateway delivers the event to all subscribers.

The Node.js SDK's network.getChaincodeEvents() returns an async iterable — we loop over it forever, receiving each event as it arrives. This page streams those events to your browser in real-time using Server-Sent Events (SSE).

The immutable audit trail
💡 Immutability of the blockchain

Every submit() call you make is permanently recorded as a transaction in a block. Even if an asset is later deleted, the transaction that created it still lives in the blockchain forever. This makes Fabric ideal for auditing — you can always prove what happened and when.

About Fabric Block Structure

In Hyperledger Fabric, the ledger is made up of two components:

1. World State — A key-value database (LevelDB or CouchDB) holding the current value of each asset. When you call evaluateTransaction('GetAllAssets'), you're reading from here.

2. Blockchain (Transaction Log) — An append-only sequence of blocks, each containing a batch of transactions. This is the immutable audit trail. Deleting an asset only removes it from World State — the creation transaction is still in the chain.

Every transaction in a block contains: the read set (what data was read), the write set (what data was written), the endorsers' signatures, and a hash linking it to the previous block.
